什麼是 Vibe Coding？

Vibe Coding 是 Andrej Karpathy 在 2025 年提出的開發哲學：完全沉浸在用 AI 描述意圖、不糾結程式碼細節的開發流程。

💬 Karpathy 原話：「There's a new kind of coding I call 'vibe coding', where you fully give in to the vibes, embrace exponentials, and forget that the code even exists.」—— 意即：放棄對程式碼的控制感，完全由 AI 驅動，只要描述你想要什麼。

✨ Vibe Coding 的核心精神

  • 用自然語言描述需求，AI 負責實作
  • 不逐行審查程式碼，只驗收成果
  • 快速迭代——看到問題就截圖/描述，讓 AI 修
  • 把精力放在「要做什麼」，不是「怎麼做」
  • 接受 AI 產生的程式碼，甚至不完全理解它
  • 以對話為主要開發介面

🎯 Vibe Coding 最適合

  • 個人 SaaS MVP / 副業項目
  • 快速原型驗證商業想法
  • 內部工具、自動化腳本
  • 前端 Landing Page / Demo
  • 非技術人員想實現想法
  • 小型獨立開發者（Indie Hacker）

⚠️ 不適合 Vibe Coding 的場景

  • 金融、醫療等高合規需求系統
  • 大型複雜 codebase（>10 萬行）
  • 需要深度安全審計的應用
  • 效能極度敏感的底層系統
  • 多人長期協作維護的專案
  • 有嚴格程式碼審查流程的企業

概念釐清

Vibe Coding vs AI Agent

這兩個概念常被混用，但有本質差異。了解差異，才能選對工具。

⚡ Vibe Coding

  • 以人為主導，AI 負責執行
  • 對話式互動，一問一答
  • 人類決定每一步要做什麼
  • 適合：程式碼生成、Debug、解釋
  • 代表工具：Cursor Chat、Claude
  • 風險：低，可逐步驗證
  • 自主程度：低～中
  • 最適合：學習、探索、新功能開發
VS

🤖 AI Agent 模式

  • AI 自主規劃並執行多步任務
  • 一次給目標，AI 決定路徑
  • 可同時修改多個文件、執行指令
  • 適合：大型重構、跨文件任務
  • 代表工具：Cursor Composer、Cascade
  • 風險：中高，需仔細審查
  • 自主程度：高
  • 最適合：已明確需求的批量任務

💡 實務建議：大多數情況下，Vibe Coding 與 AI Agent 並非二選一。最佳實踐是：用 Vibe Coding 探索與設計，再用 Agent 模式批量執行確定好的任務。例如：先對話釐清需求，再讓 Cursor Composer 自動實作整個功能模組。

短期內不會，但工程師的角色確實在轉變。Vibe Coding 大幅降低了初級程式開發的門檻，但系統設計、架構決策、效能優化、安全審查等高階工程能力需求反而增加。未來更需要「能看懂、能審查、能導向 AI 輸出」的工程師，而不只是能打字寫程式的人。掌握 AI 工具本身就是一種核心競爭力。

Kiro 是 Amazon AWS 在 2025 年推出的 AI IDE，最大特色是「Spec-First」工作流：不是直接生成程式碼，而是先幫你生成 PRD（產品需求文件）→ 架構設計 → 任務清單，再逐步實作。這讓開發過程更有結構，特別適合需要文件完整性的企業環境。而 Cursor 更偏向對話式、即時生成，適合快速迭代。兩者可互補使用：用 Kiro 做規劃，用 Cursor 做實作。
