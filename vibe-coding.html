AI 輔助開發的
真實樣貌

Vibe Coding 不是把需求丟給 ChatGPT 複製貼上。
它是一個持續的開發迴圈——AI 理解你的程式架構、讀取錯誤訊息、跨檔案重構，更像是一位隨時在線的 pair programmer。

往下滑
認識概念

Vibe Coding ≠ 用 AI 聊天

一般人對 AI 寫程式最大的誤解：以為 Vibe Coding 就是把問題貼給 ChatGPT。這兩件事在技術本質上截然不同。

💬 普通 AI 對話（Stateless Prompting）

✗ 每次對話都是獨立的，AI 不記得你上一個 session 的程式碼
✗ AI 只看見你貼進去的片段，不知道其他檔案的內容
✗ 沒有程式碼執行能力，無法驗證生成的程式是否真的能跑
✗ 修改一個地方，不知道有沒有影響其他檔案的依賴關係
✗ 每次都要重新解釋背景，效率極低

⚡ Vibe Coding（Contextual Dev Loop）

✓ 工具索引整個 codebase，AI 理解你的資料庫結構、import 關係、型別定義
✓ 用 @ 符號直接引用特定檔案，AI 看得到完整的上下文
✓ 整合終端機，可以執行程式、讀取 stack trace，自動找出錯誤根因
✓ 跨檔案重構：改一個函式名稱，自動找到所有呼叫它的地方一起改
✓ 持續的開發脈絡，AI 知道你的專案現在在哪個階段

技術深度

Context Window 與 Codebase 感知

Vibe Coding 工具的關鍵技術：讓 AI 真正「看懂」你的整個專案，而不只是你貼給它的片段

📁 Cursor 的 codebase 索引
📂 my-app/
📂 src/
📄 @api/users.ts  ← 你可以直接 @ 引用
📂 components/
📄 UserProfile.tsx  ← 目前編輯的檔案
📄 UserSkeleton.tsx
📂 hooks/
📄 useUser.ts
📂 types/
📄 user.d.ts  ← AI 知道你的型別
📄 package.json
📄 tsconfig.json
🔍 Cursor 做了什麼？
它用 embedding 把整個 codebase 向量化，當你提問時，自動找出最相關的程式片段放進 context window，讓 AI 生成的程式碼能無縫整合你既有的架構，不會憑空發明一個和你命名習慣不同的函式。
💬 @ 引用的實際用法
你輸入的 Prompt：
「看一下 @types/user.d.ts 和 @api/users.ts，在 UserProfile 元件裡幫我加一個可以更新 displayName 的 inline edit 功能，要和現有的 useUser hook 整合」
AI 看到的 context 包含：
✓ User 型別的所有欄位定義
✓ API 的 endpoint 路徑與回傳格式
✓ useUser hook 的介面與回傳值
✓ 你的 TypeScript strict 設定
✓ 你既有的 import 路徑慣例
ChatGPT 對話框 vs. 這個差在哪：
ChatGPT 只看到你貼進去的那幾行，
容易產生和你真實架構不相容的程式碼，
你還得手動調整 import 路徑和型別名稱。
📖 Context Window 的限制
即使是最好的工具，context window 也有上限（通常 200k tokens）。超大型 codebase 中，AI 可能無法一次看到所有相關程式碼。Checkpoint commit + 定期清理對話脈絡是避免 context 汙染的關鍵。

⚠ 錯誤的實際樣貌

AI 把完整的 stack trace 讀進 context，定位到是第幾行哪個函式出錯，而不是靠你描述「好像第 23 行有問題」

TypeError: Cannot read properties of undefined
  at UserProfile (UserProfile.tsx:23:15)
  at renderWithHooks (react-dom.js:14985)

→ AI 診斷：useUser hook 在 data 為 undefined 時
  沒有 null guard，在第 23 行直接取 data.name
  已在 optional chaining 加上修正：data?.name

✓ 這個迴圈多快？

有經驗的 Vibe Coders 一個迴圈只需要 30 秒到 2 分鐘。一小時內可以迭代幾十次，相當於傳統開發一整天的 debug 週期。

⏱ 平均 30-90 分鐘

# Vibe Coding
寫程式 → 跑 → AI 讀 trace → 修好
⏱ 平均 1-3 分鐘

💡 Checkpoint 策略

每完成一個可運行的功能就 commit 一次。這不只是版本控制，而是 Vibe Coding 的安全網——AI 做大型重構時，你可以隨時回到上一個穩定版本。

git commit -m "feat: add user skeleton"
# ← 這是你的 checkpoint

# 接下來讓 AI 做大改動
# 如果搞砸了：
git reset --hard HEAD
# 回到安全點，重新下 prompt

🎯 「解釋你改了什麼」的重要性

讓 AI 解釋每次修改的理由，是 Vibe Coding 新手最容易忽略的技巧——不只是接受結果，而是理解決策。

你：「改完後解釋一下
   你改了哪些檔案、為什麼」

AI：修改了 2 個檔案：
1. UserProfile.tsx - 加 optional chaining
2. useUser.ts - 加 initialData 預設值
原因：避免 race condition...

Prompt 技巧

Vibe Coding 專屬的 Prompting 模式

這不是通用的「如何問 AI 問題」——這些是 Vibe Coding 開發流程中，讓迭代效率倍增的具體技巧

Pattern 01
先給架構脈絡，再提功能需求
先讓 AI 理解你的技術棧和架構決策，之後每個 prompt 才不需要重新解釋背景，AI 生成的程式碼也更貼合你的專案。
✗ 直接問功能
「幫我做一個登入頁面」
✅ 先建立脈絡
「這個專案用 Next.js 14 App Router、TypeScript、Prisma + PostgreSQL、NextAuth。驗證策略是 email/password + Google OAuth。了解後幫我做登入頁面，要整合現有的 auth.ts 設定。」
Pattern 02
要求增量修改，而不是重寫
要求 AI 在現有程式碼上加功能，而不是從頭重寫。這樣可以保留你手動調整過的部分，避免 AI 做出過度的改動，難以追蹤變化。
✗ 容易造成大幅改動
「幫我重寫這個元件，加上 loading 狀態」
✅ 指定增量修改
「在現有的 UserList 元件裡，只加一個 isLoading prop，當它為 true 時顯示 3 個 skeleton 行。不要改其他邏輯。」
Pattern 03
把大功能拆解成 AI 可執行的子任務
「幫我做一個電商網站」對 AI 來說太模糊。把它拆成每次一個可驗證的子任務，AI 完成一步，你確認沒問題再繼續。
✗ 太大一口
「幫我做一個有購物車、結帳、會員系統的電商網站」
✅ 拆解成步驟
Step 1：「建立商品資料的 Prisma schema」
Step 2：「建立取得商品列表的 API route」
Step 3：「做商品卡片 UI 元件」
（每步驟確認後再繼續）
Pattern 04
要求 AI 同步寫測試
要求 AI 在生成功能的同時寫單元測試，這能讓 AI 自我驗證邏輯，也讓你的反饋迴圈更快——跑測試比手動點 UI 快 10 倍。
✗ 只要功能
「幫我寫一個計算折扣的函式」
✅ 功能 + 測試同步
「寫一個 calculateDiscount 函式，處理百分比折扣和固定金額折扣，同時用 vitest 寫至少 5 個測試案例，包含邊界條件（負數、超過 100%）」
Pattern 05
大改動前，先讓 AI 說計畫
要求 AI 在執行之前先列出它打算修改哪些檔案、做什麼改動。你先審查計畫，沒問題再讓它執行，避免 AI 偷偷改了你不想動的地方。
✗ 直接讓 AI 動手
「幫我把整個認證系統從 session 改成 JWT」
✅ 先要計畫再執行
「我要把認證從 session 改成 JWT。先不要改任何程式碼，列出你打算修改哪些檔案、每個檔案改什麼，我確認後再開始。」
Pattern 06
保持對話中的「理解確認」
定期問 AI「你對這個功能的理解是什麼？」可以早期發現 AI 是否誤解了你的需求，避免走偏後才發現，省下大量修改時間。
✗ 假設 AI 理解正確
（直接讓 AI 開始做，最後發現方向完全不對）
✅ 先確認理解
「在開始之前，用一段話告訴我你對這個功能的理解，以及你打算用什麼技術方案，為什麼。」
誠實面對

Vibe Coding 什麼時候會失靈

不談限制的教學是不完整的。理解 AI 的邊界，才能讓你在關鍵時刻做出正確的判斷

👻
幻覺 API 和不存在的函式
AI 有時會自信地呼叫一個根本不存在的函式或 library，尤其是較新版本的套件。這個錯誤不會有語法錯誤提示，跑起來才爆。
✅ 對策
要求 AI 標出它用了哪些外部套件，跑前先查 npm/docs 確認 API 名稱正確
🌊
跨檔案錯誤的累積效應
AI 修了 A 檔案，卻沒注意到 B 和 C 也依賴相同的邏輯。多次迭代後，這些小不一致性層層累積，最後整個 codebase 狀態難以理解。
✅ 對策
每次重構後讓 AI 做「影響範圍分析」，並在修改後跑全部測試
🌫️
長對話中的 Context 汙染
對話越長，AI 越可能把早期的舊資訊和新需求混在一起，開始產生自相矛盾的程式碼。這是 context window 的固有限制。
✅ 對策
完成一個功能 checkpoint 後，開啟新的對話，貼入最新的相關程式碼重新開始
🔐
安全盲點
AI 生成的程式碼可能缺乏 SQL Injection 防護、把 API key 硬寫在程式碼裡、或忘記 rate limiting。AI 優化的是「能跑」，不是「安全」。
✅ 對策
涉及用戶資料或外部 API 的功能，明確要求 AI 做安全審查並解釋防護措施
🐢
架構債的快速累積
AI 習慣用最直接的方式解決當前問題，不會主動考慮長期可維護性。快速迭代 10 個功能後，codebase 可能已經難以繼續擴展。
✅ 對策
每隔幾個功能，讓 AI 做一次「程式碼品質審查」，主動指出重構點
🎲
不懂程式碼就無法把關
完全不懂程式的人做 Vibe Coding，很難判斷 AI 的輸出是否合理。遇到 AI 信心滿滿但其實是錯的情況，你可能完全不知道。
✅ 對策
用「解釋給我聽」模式：讓 AI 用白話解釋它的每個決定，培養基本的判斷能力
誠實說：Vibe Coding 不能取代工程師的判斷力。
它是一個威力強大的加速器，但你仍然需要理解「什麼叫做好的架構」、「什麼是安全問題」、「什麼時候該停下來做正確的設計」。懂得這些邊界，才能讓 Vibe Coding 發揮最大價值。
入門教學

從零建立你的第一個 Vibe Coding 工作流

這不是「試試看貼個 prompt」的教學——這是建立一個能持續迭代的開發閉環

1

安裝 Cursor，建立第一個專案

下載並安裝 Cursor（免費版夠用），用它開啟一個資料庫。Cursor 會自動索引你的檔案結構。如果是全新專案，先讓 AI 幫你建立基礎架構。

💬 初始架構 Prompt
「幫我建立一個 Next.js 14 App Router 專案，用 TypeScript、Tailwind CSS、shadcn/ui。建立基本的資料庫結構和必要的設定檔，並解釋每個資料庫的用途。」
2

建立「架構文件」讓 AI 持續理解你的專案

在專案根目錄建立一個 CONTEXT.md，記錄你的技術棧、架構決策和命名慣例。每次開新對話時，先 @-引用這個檔案。

📄 CONTEXT.md 模板
「# 專案：MyApp
技術棧：Next.js 14 / TypeScript / Prisma / PostgreSQL
UI：shadcn/ui + Tailwind
認證：NextAuth v5
命名：元件用 PascalCase，hooks 用 useXxx，API routes 在 /app/api/...」
3

用「功能 → 執行 → 確認」的迴圈開發

每次只做一個小功能，做完就跑起來確認，沒問題就 commit。不要讓 AI 一次做太多事情——功能越小，問題越容易定位。

⚡ 關鍵心態：你是 PM，AI 是工程師。你的工作是把需求切成夠小的任務，讓 AI 能一步步完成，而不是丟一個大需求等 AI 神奇地幫你全做好。
4

遇到錯誤：讓 AI 讀 Stack Trace，不要自己解讀

程式跑錯了？把整個錯誤訊息（包含完整 stack trace）貼給 AI，同時 @-引用相關檔案。讓 AI 定位根因，不要只說「它壞掉了」。

✗ 效率低的說法
「它報錯了，幫我修一下」
✅ 高效的錯誤回報
「跑 @UserProfile.tsx 時出現這個錯誤：[貼上完整 stack trace]。@hooks/useUser.ts 也一起看一下，找出根因並修復。」

養成 Checkpoint Commit 習慣

每完成一個可運行的功能，立刻 commit。這是 Vibe Coding 最重要的紀律——讓你可以放心地讓 AI 做大改動，知道任何時候都能回頭。

🎯 判斷標準：「如果 AI 接下來的修改完全搞砸了，我能在 30 秒內回到這個狀態嗎？」能的話才繼續。

常見問題

你可能想知道的事

Vibe Coding 和普通 Prompt 對話差在哪？一句話說明。 +
普通 Prompt 是無狀態的一次性問答（stateless Q&A），每次對話都是從零開始，AI 不知道你的專案長什麼樣。Vibe Coding 是有狀態的持續開發迴圈——AI 索引了你的 codebase、能讀取執行錯誤、能跨檔案修改，更像是一位了解你整個專案的 pair programmer。
不懂程式的人可以學 Vibe Coding 嗎？ +
可以，但有一個誠實的限制：你越懂程式，Vibe Coding 的效果越好。完全不懂程式的人可以做出簡單的工具，但遇到複雜問題時可能難以判斷 AI 的輸出是否正確。建議同步學習基礎的 HTML/CSS/JavaScript 概念，即使只是閱讀層面的理解，也能讓你的 Vibe Coding 品質大幅提升。
AI 有時候改了一個地方，另一個地方就壞掉，怎麼辦？ +
這是 Vibe Coding 中最常見的問題，叫做「跨檔案一致性問題」。解法是：① 在 prompt 中加入「確認這個修改不會影響其他呼叫這個函式的地方」；② 要求 AI 在修改後列出「受影響的檔案清單」；③ 養成寫測試的習慣，讓 AI 改完後跑測試確認沒有破壞其他功能。
Cursor 和 Claude 在使用上有什麼本質差異？ +
Cursor 是整合在編輯器裡的開發環境，它有完整的 codebase 索引、Terminal 整合和跨檔案感知——這是真正的 Vibe Coding 體驗。Claude 是網頁對話介面，沒有 codebase 索引，你需要手動貼程式碼，但它的 200k token context window 很長，適合分析大型單一檔案或做架構討論。兩者可以配合使用：Cursor 用來日常開發，Claude 用來做大局分析和架構設計。
AI 生成的程式碼有安全問題嗎？ +
確實存在安全盲點。AI 優化的是「功能能跑」，不一定會主動考慮 SQL Injection 防護、XSS、API key 洩漏等安全問題。解法：① 涉及用戶資料的功能，明確要求「做完後幫我做安全審查」；② 永遠不要把 API key 或密碼直接寫在程式碼裡，要用環境變數；③ 生產環境的程式碼，找有安全背景的人做 code review。
Vibe Coding 做出來的東西，能到什麼規模？ +
個人工具、Landing Page、MVP 產品、中型 SaaS——都有成功案例。目前的限制主要在於：超過幾萬行的大型 codebase 管理困難（context window 不夠放），以及複雜的分散式系統架構還是需要有經驗的工程師設計。但對 90% 的個人專案和小團隊需求來說，Vibe Coding 已經相當夠用，而且速度遠比傳統開發快。
